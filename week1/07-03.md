### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: ContentType 도입과정에 있어서 if else -> helper method -> ENUM 도입으로 MIME 타입 판단 책임 분리
- RequestHandler의 ContentType 판단 책임 점진적 개선을 위해 초기에는 if else 로 판단했습니다.
- 여기까지만 해도 주어진 요구사항의 기능은 동작하였으나 새로운 MIME 추가될 때마다 file 수정이 필요했고
- 이는 MIME 추가가 Handler 의 수정으로 이어지기에 OCP 위배라고 생각했습니다. 또 더해서 HTTP 개념이 Handler 에서 생겼기에
- 헬퍼 메소드로 분리했지만, OCP는 마찬가지여서 이후 다른 객체로 빼서 책임을 나누고 ENUM으로 타입 안전성을 갖고 확장자별 MIME 타입 매핑을 응집시켰습니다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 왜 accept vs 왜 확장자?
- 왜 확장자냐면 정적 (JS CSS HTML SVG)등을 요청하기 때문에 서버입장에서는 이미 URL에 담겨져있는 요청을 판단하는게 더 적절하다고 생각
- 근데 나중에 accept를 뜯어야될 동적 요청이 온다면을 위해서 headers 를 list 로 만들어서 저장해뒀다. (while 으로 read 해서 add.)
