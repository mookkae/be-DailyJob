### 2. 오늘 가장 의미있었던 기술적 판단

- 고민의 흔적: 고객과 배송지라는 테이블에 있어서 기본 배송지라는 속성을 어떻게 다뤄야될지 고민했다.
- 회원가입 시 기본 배송지를 입력하니 customer의 속성으로 볼지 아니면 address 로써 다룬다면 customer_id 를 PK 로 둘지
- 그래서 나는 후자의 결정을 정했다.
- 그런다음 기본 배송지의 여부에 대해서 boolean type 으로 기본배송지를 표시했다.
- 왜 customer 의 속성으로 안했는지에 대해서 서술해보자면?

1. 확장성을 고려하자면 가입시점에는 주소가 1:1 로 볼 수 있지만 어쨋든 더 배송지를 추가한다는 확장성
2. 테이블의 관심사의 분리 SoC 고객이라는 테이블이 관리할만한 주체가 주소가 1개라면 이해되지만 여러개의 주소를 다 가지고 있을 필요는 없다고 판단.
3. 무결성 고객의 주소가 삭제될 때 양쪽 테이블을 다 맞춰줘야 하는 수고

### 3. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 어떤 행동이었는지: DB 설계 어색
- 백엔드 개발자임에도 불구하고 데이터를 개념적에서 그치고 논리적으로 이어지는것에 대해서 부족하다.
- 이는 JPA 나 AI 발전에 있어서 ERD 나 관계를 작성하는것에 trade off 만 제시하며 AI에게 위임했기 때문에 발생했다 생각한다.
- DBA 라는 직무가 아니더라도 직접적으로 DB 를 다루기 때문에 SQL 을 손으로 짜는것과 더해서 ERD 를 구성하고 Relation 을 어떻게 맺는지도 판단했어야 했다.
- 주문과 상품의 경우 N : M 이지만 이를 분리하여 관계형 테이블로 주문과 그에 대한 상품을 기록하는 주문 상품 테이블로 빼서 기존 주문과 상품의 row 에 영향 없게 한다.
- 왜? 관계를 풀어내야할까?

1. RDB 는 하나의 컬럼에 여러개 저장못함 제1정규형 고려
2. 중간 테이블 없이 하면 이상현상 발생가능 주문 1개에 상품이 3개일때 똑같은 주문정보가 3번 저장될 수 있는 우려.
3. 매핑 뿐만아니라 비즈니스 도메인으로 과거의 기록을 보존 가격이나 변경
